# **Render Migration & Backend Multi-Container Setup**

*Migrate from Heroku to Render using Docker (NestJS Monorepo)*

## **🧭 Context**

We are migrating **all backend services** from Heroku to **Render**.

> ✅ This migration uses **ONE Render app** for all microservices, each
> running in a separate Docker container.\
> ❌ Do NOT create multiple Render apps (avoids billing confusion and
> dev errors).\
> ✅ This guide assumes the monorepo is already structured under
> marketeq-projects-nestjs.

## **🧱 Monorepo Folder Structure**

The base structure inside the marketeq-projects-nestjs repo looks like
this:

marketeq-projects-nestjs/

├── apps/

│ ├── user-service/

│ ├── auth-service/

│ ├── review-service/

│ ├── listing-service/

│ ├── messaging-service/

│ └── \...

├── docker/

│ ├── user-service.Dockerfile

│ ├── auth-service.Dockerfile

│ └── \...

├── .github/

│ └── workflows/

├── render.yaml

└── \...

Each service has its own Dockerfile inside the /docker directory.

## **🐳 Step-by-Step Migration to Render (Single App, Multi-Container)**

### **1. Prepare render.yaml**

This file defines all services inside your single Render app.

services:

\- type: web

name: user-service

env: docker

dockerfilePath: ./docker/user-service.Dockerfile

buildCommand: \"\"

startCommand: \"npm run start:prod\"

envVars:

\- key: DATABASE_URL

value: YOUR_POSTGRES_URL

\- key: JWT_SECRET

value: your_jwt_secret

\- type: web

name: auth-service

env: docker

dockerfilePath: ./docker/auth-service.Dockerfile

startCommand: \"npm run start:prod\"

envVars:

\- key: DATABASE_URL

value: YOUR_POSTGRES_URL

\- key: JWT_SECRET

value: your_jwt_secret

\# Add more services as needed

> ✅ Define **all containers in one render.yaml file**.\
> 📝 Add any additional .env variables per service.

### **2. Connect GitHub Repo to Render**

- Log into [[Render\]{.underline}
  ](https://dashboard.render.com/)

- Click **\"New Web Service\"\**

- Choose **\"From YAML\"\**

- Connect the GitHub repo: marketeq-projects-nestjs

- Select the render.yaml file

- Confirm deployment settings

> 🔁 Render will auto-deploy on every push to the staging or live
> branch.

### **3. CI/CD Integration via GitHub Actions**

Use backend.yml in .github/workflows/:

name: CI for Backend

on:

push:

branches:

\- staging

\- live

jobs:

build:

runs-on: ubuntu-latest

steps:

\- name: Checkout code

uses: actions/checkout@v3

\- name: Set up Node.js

uses: actions/setup-node@v3

with:

node-version: \'18\'

\- name: Install deps

run: npm ci

\- name: Run Tests

run: npm run test

\- name: Deploy to Render

run: echo \"Render deploy happens automatically via repo sync.\"

> 🛑 Do **not** manually deploy via CLI unless necessary.\
> ✅ Push to staging triggers automatic build & deploy of all containers
> via render.yaml.

### **4. Post-Migration Tasks**

- Delete old Heroku apps to avoid billing

- Ensure all .env values are correctly added to each Render service

- Confirm health checks, logs, and autoscaling are working

- Add render.yaml to .github/ISSUE_TEMPLATE/ as a reference
