**CI/CD Guardrails & QA Handoff Rules**

*Marketeq Backend & Frontend \| Staging → Live Workflow (Render +
Vercel)*

## **🧠 Objective**

This doc ensures that our CI/CD workflows for **backend (Render)** and
**frontend (Vercel)** follow a consistent, automated, and traceable
structure --- so QA can test effectively and developers don't block
releases.

## **🚦 Git Branches & Deployment Environment Rules**

  ---------------------------------------------------------------------
  **Branch**    **Environment**   **Auto            **Who Can Merge?**
                                  Deployment**      
  ------------- ----------------- ----------------- -------------------
  development   Local only        ❌                Any developer

  staging       Staging           ✅                Team leads only
                                  (Render/Vercel)   

  live          Production        ✅                CTO / Release
                                  (Render/Vercel)   approver
  ---------------------------------------------------------------------

> ⚠️ Never push directly to staging or live. Use PRs only.

## **🧪 CI/CD Testing & Automation**

### **Backend CI/CD (Render via GitHub Actions)**

- All PRs into staging and live trigger:

  - Lint checks

  - Unit tests

  - Type-checking

  - Render deploy via render.yaml

- Badge status is auto-injected into README

### **Frontend CI/CD (Vercel + GitHub)**

- Vercel previews every PR.

- Push to staging = deploy to staging URL

- Push to live = deploy to production

## **🧰 Developer & QA Workflow**

### **👨‍💻 Developer Workflow**

1.  Code locally → push to development

2.  Open PR to staging (review required)

3.  Merge triggers deploy to staging (Render/Vercel)

4.  Notify QA team in Slack #qa-handoff

### **🧪 QA Workflow**

1.  Confirm staging is up (backend + frontend)

2.  Test full user flow (no broken endpoints)

3.  Mark JIRA ticket as "Ready for Production"

4.  Notify CTO or release approver

### **🚀 Final Deployment**

1.  CTO merges PR from staging → live

2.  Production is deployed (Render + Vercel)

3.  QA sanity check on live environment

## **❗ Guardrails to Prevent Developer Errors**

  **Mistake**                              **Prevention Mechanism**
  ---------------------------------------- -----------------------------------------------------------------------------------------
  Frontend-backend integration too early   Add comment in onboarding: don't test integration until backend endpoints are confirmed
  Direct push to staging/live              Enable **GitHub branch protection rules**
  Forgetting .env variables                Add checklist before PR is approved
  Deploying without QA                     Slack #qa-handoff required before release

## **🔐 GitHub Branch Protection Rules (Recommended)**

Apply to staging and live branches:

- ✅ Require pull request before merging

- ✅ Require status checks to pass (CI)

- ✅ Require linear history

- ✅ Require code review by 1 team member

- ✅ Restrict who can push

## **🧾 Summary**

- Automate everything --- no manual deploys.

- Protect staging & live branches with strict rules.

- QA team is notified automatically when staging is deployed.

- Developers can't bypass review or testing.

- One staging, one live --- nothing else.
