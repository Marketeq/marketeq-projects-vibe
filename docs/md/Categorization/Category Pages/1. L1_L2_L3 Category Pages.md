# **Doc 1 --- L1/L2/L3 Category Pages**

**Architecture, URL Scheme, and Page Payloads (Show-All + Per-Entity
Pages)**

Goal: Deliver copy-paste production code for routing, level detection,
canonical URLs, SSR, and a single API that returns the **page payload**
(Show-All + entity Pages) for any L1/L2/L3 category path. Steps include
explanation, folder path, and working code.

## **Step 1 --- Define URL strategy & level detection**

**Explanation\**
We unify all category URLs under /c/:l1, /c/:l1/:l2, /c/:l1/:l2/:l3.

- **L1 pages** = Top-level categories.

- **L2 pages** = Subcategories of L1.

- **L3 pages** = Subcategories of L2.

We detect the level by the number of URL segments and resolve to the
correct category UUIDs using the category tree. Breadcrumbs are built
recursively (L1 → L2 → L3).

Inside each page, you can still have **tabs like "Most Popular" or "Most
Recent"** --- but those are just different *views of the same page*, not
different category levels.

**Folder path\**
/apps/web/pages/c/\[\...slug\].tsx

export const getServerSideProps: GetServerSideProps = async (ctx) =\> {

const slug = (ctx.params?.slug as string\[\]) \|\| \[\]

const viewParam = (ctx.query?.view as string) \|\| null

const url =
\`\${process.env.NEXT_PUBLIC_API_URL}/category-pages/\${slug.join(\'/\')}\${viewParam
? \`?view=\${viewParam}\` : \'\'}\`

const res = await fetch(url, { headers: { \'x-edge-cache\': \'allow\' }
})

if (res.status === 404) return { notFound: true }

const payload: PagePayload = await res.json()

// Tab URLs are exposed as path segments, not ?view=.

!ctx.resolvedUrl.endsWith(\`/\${payload.view}\`)) {

return {

redirect: {

destination: \`/c/\${payload.slugPath.join(\'/\')}/\${payload.view}\`,

permanent: true

}

}

}

return { props: { payload } }

## **Step 2 --- Page Payload API (Assembler) in NestJS**

**Explanation\**
We expose **one API** that:

1.  resolves the category by slug path (L1/L2/L3),

2.  detects the level and returns breadcrumbs & subcategories,

3.  assembles the **Show-All view** or an **entity-specific view
    (Projects, Services, Teams, Talent)** within the same category page
    using rules,

4.  applies **Manual Overrides** if enabled (pinned first, AI/rule
    results fill the rest),

5.  picks **grid vs carousel** per design flags,

6.  enforces **carousel min content (3× grid)** and **max 10 scrolls**,

7.  uses existing SEO defaults/overrides.

**Folder paths**

- Controller: /apps/api/src/category-pages/category-pages.controller.ts

- Service: /apps/api/src/category-pages/category-pages.service.ts

- Assembler: /apps/api/src/category-pages/assembler.ts

- Evaluators: /apps/api/src/category-pages/evaluators/\*.ts

- Types: /apps/api/src/category-pages/types.ts

**Code (Controller)**

****// /apps/api/src/category-pages/category-pages.controller.ts

import { Controller, Get, Param, Query, Res } from \'@nestjs/common\'

import { Response } from \'express\'

import { CategoryPagesService } from \'./category-pages.service\'

\@Controller(\'category-pages\')

export class CategoryPagesController {

constructor(private readonly svc: CategoryPagesService) {}

\@Get(\'\*\')

async resolve(@Param() \_p: any, \@Query(\'view\') view: string, \@Res()
res: Response) {

const path = (res.req.path.replace(/\^\\/category-pages\\//, \'\') \|\|
\'\').split(\'/\').filter(Boolean)

const last = path\[path.length - 1\]

const knownTabs = new
Set(\[\'projects\',\'services\',\'teams\',\'talent\'\])

const tab = knownTabs.has(last) ? (last as any) : (view as any) \|\|
\'show-all\'

const slugPath = knownTabs.has(last) ? path.slice(0, -1) : path

const payload = await this.svc.buildPagePayload(slugPath, tab)

if (!payload) return res.status(404).send({ error: \'Category not found
or unpublished\' })

return res.json(payload)

}

}

private readonly taxGroups: TaxGroupRepo,

private readonly assembler: Assembler

async buildPagePayload(slugPath: string\[\], view: TabView):
Promise\<PagePayload \| null\> {

// 1) Resolve category level and UUIDs (L1/L2/L3) and ensure published
gate

const resolved = await this.categories.resolvePath(slugPath) // returns
{ level, uuids, breadcrumbs, published }

if (!resolved \|\| !resolved.published) return null

// 2) Gather subcategories for the current node (facets reflect
parentage L1→L2→L3)

const subcategories = await this.categories.childrenOf(resolved.uuids\[
\`l\${resolved.level}\` \])

// 3) Load applicable Tax Groups for this level + path

const rules = await this.taxGroups.forPath(resolved.uuids,
resolved.level)

// 4) Assemble Show-All or entity-page using rules (+ Manual Overrides
if enabled)

const blocks = await this.assembler.composeBlocks({

level: resolved.level,

uuids: resolved.uuids,

view,

rules

})

// 5) SEO from existing hybrid logic

const seo = await getSeoForPath(slugPath) // uses overrides when present

return {

level: resolved.level,

slugPath,

uuids: resolved.uuids,

breadcrumbs: resolved.breadcrumbs,

seo,

view,

blocks,

subcategories

}

}

// /apps/api/src/category-pages/assembler.ts

import { Injectable } from \'@nestjs/common\'

import { EvaluatorFactory } from \'./evaluators/factory\'

import { ManualOverrideRepo } from \'../repos/manual-override.repo\'

import { BlockSpec, ComposeArgs } from \'./types\'

const MAX_CAROUSEL_PAGES = 10

\@Injectable()

export class Assembler {

constructor(

private readonly factory: EvaluatorFactory,

private readonly overrides: ManualOverrideRepo

) {}

async composeBlocks(args: ComposeArgs): Promise\<BlockSpec\[\]\> {

const { level, uuids, view, rules } = args

const specs: BlockSpec\[\] = \[\]

// Decide which blocks render based on design flags attached to rules
(carousel vs grid)

for (const rule of rules) {

if (view !== \'show-all\' && rule.entity !== view) continue

const evaluator = this.factory.for(rule.entity, rule.kind)

const minForCarousel = rule.carousel ? rule.gridSize \* 3 : 0

// Hybrid ordering: manual pins first (if enabled), then AI/rule-based
fill

const manual = rule.manualEnabled

? await this.overrides.pinnedItems(rule.id, uuids, rule.limit)

: \[\]

const needed = Math.max(0, rule.limit - manual.length)

const aiResults = needed \> 0

? await evaluator.run({ uuids, level, rule, limit: needed })

: \[\]

const items = \[\...manual, \...aiResults\]

// Enforce carousel policy

if (rule.carousel) {

if (items.length \< minForCarousel) continue

const perSlide = rule.gridSize

const maxItems = perSlide \* MAX_CAROUSEL_PAGES

specs.push({

key: \`rule-\${rule.id}\`,

type: \'carousel\',

entity: rule.entity,

items: items.slice(0, maxItems)

})

} else {

specs.push({

key: \`rule-\${rule.id}\`,

type: \'grid\',

entity: rule.entity,

items

})

}

}

return specs

}

}

## **Step 3 --- Rule evaluators (entities + ranking blend)**

**Explanation\**
Each entity (Projects/Services/Teams/Talent) plugs into a **Rule
Evaluator** with a common interface. Evaluators respect **subtree
scopes** (L1→L2→L3 parentage), **moderation/publishing flags**, and the
**ranking blend** (recency, rating, completion count, revenue, CTR,
boosts). We'll later add the Hugging Face model to the blend; for v1
we'll include the HF inference hook so we won't rewrite code later.

**Why subtree queries?** Because L2 is a child of L1 and L3 is a child
of L2; the evaluator must filter by the current **node's subtree** to
keep categories and results accurate.

**Folder path\**
/apps/api/src/category-pages/evaluators/{projects,services,teams,talent}.ts

## **Step 5 --- Facets/filters respect parentage (L1→L2→L3)**

**Explanation\**
Facets are derived **from category hierarchy**: L1 provides top-level
category facets; L2 narrows to its children; L3 tightens to
grandchildren. We're **not** introducing budget bands/team size at this
time per your direction; facets mirror the taxonomy, autocomplete
signals, and design.

## **Step 7 --- Caching & revalidation**

**Explanation**

- **SSR** for HTML (Next.js), cached at the edge (CDN).

- **API cache keys** include: level, uuids, view, **TaxGroup hash**, and
  **override version**.

- **Invalidate** on Strapi webhooks (category/tag/tax-group changes)
  with cron fallback (you already have both patterns).

## **Step 8 --- Error & empty-state handling (MVP vs GA)**

**Explanation**

- **Unpublished categories** → 404 to public, but accessible via admin
  preview; still present in autocomplete.

- **Temporary emptiness (alpha/beta)** → we allow emptiness during
  internal builds; for GA we enforce thresholds and simply keep category
  unpublished until it's "full."
