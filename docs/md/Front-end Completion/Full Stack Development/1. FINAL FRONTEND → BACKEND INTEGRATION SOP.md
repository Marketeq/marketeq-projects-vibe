# **FRONTEND → BACKEND INTEGRATION SOP**

*(Generalized procedure using "Favorites" screen as an example)*

## **Purpose**

This document provides a complete step-by-step guide for integrating any
frontend screen with its corresponding backend microservice.\
\
The **Favorites** screen is used only as an example, but the same
process applies to all screens and services.

The purpose of this SOP is to ensure consistent frontend integrations,
predictable folder structures, clean code organization, and accurate
communication between the frontend and backend using approved Marketeq
standards.

## **Outcome**

By following this SOP, the full-stack developer will:

1.  Connect a Next.js frontend screen to its backend service using the
    shared HTTP client.

2.  Configure the environment variables properly based on domain
    readiness.

3.  Usetanstack query for data management and caching.

4.  Set up service-specific API modules and hooks.

5.  Implement approved loading, error, and confirmation UI components
    based on the official design system.

6.  Verify that all backend endpoints are connected correctly without
    manual guesswork.

## **Step 0 --- Verify Access and Setup**

Before starting any integration, confirm the following:

1.  You can run the Next.js app locally (Node 18 + required).

2.  You have access to the backend dev or staging API URL.

3.  The backend service has CORS enabled for your local origin
    (http://localhost:3000).

4.  If authentication is required, confirm how the token is obtained
    (NextAuth, JWT, or cookies).

## **Step 1 --- Environment Variables**

**Goal:** Ensure the frontend app points to the correct backend domain
or temporary deployment.

**Instructions**

1.  Check whether .env.example and .env.local exist in the project root.

2.  If not, create both files and add the following lines:

# .env.example

NEXT_PUBLIC_API_BASE_URL=https://projects.marketeqdigital.com

NEXT_PUBLIC_ENV=development

\# .env.local

NEXT_PUBLIC_API_BASE_URL=https://projects.marketeqdigital.com

NEXT_PUBLIC_ENV=development

3.  If the projects.marketeqdigital.com domain is not yet available:\
     • Go to the **Vercel dashboard → Settings → Environment
    Variables.\**
     • Find the temporary frontend deployment base URL and use it in
    .env.local.

4.  Restart the frontend server after saving changes.

**Important:** When the official domain is live, replace the temporary
URL with https://projects.marketeqdigital.com.

## **Step 4 --- Shared HTTP Client**

**Goal:** Provide one consistent API client for all services.

**Instructions**

1.  Check whether src/lib/http.ts already exists.\
     • If it exists, use it.\
     • If it does not exist, create it once using the template below.

2.  The base URL should match the environment variable defined earlier.\
     • Use https://projects.marketeqdigital.com once available.\
     • If not, use the temporary Vercel domain already set in
    .env.local.

3.  Token retrieval (getToken) should be implemented once globally and
    shared across screens.

4.  Do not duplicate this file or create service-specific versions.

5.  This step is done once per application and should not be repeated
    per screen.

**File:** src/lib/http.ts

export const HL = \<T,\>(v: T) =\> v

export type HttpConfig = {

baseURL?: string

getToken?: () =\> Promise\<string \| null\> \| string \| null

}

const defaultConfig: HttpConfig = {

baseURL: HL(process.env.NEXT_PUBLIC_API_BASE_URL ??
\"https://projects.marketeqdigital.com\"),

}

export function createHttp(cfg: HttpConfig = defaultConfig) {

const baseURL = cfg.baseURL ?? defaultConfig.baseURL

const getToken = cfg.getToken ?? (async () =\> HL\<string \|
null\>(null))

async function request\<T\>(path: string, init?: RequestInit):
Promise\<T\> {

const headers = new Headers(init?.headers)

headers.set(\"Content-Type\", \"application/json\")

const token = await getToken()

if (token) headers.set(\"Authorization\", \`Bearer \${token}\`)

const url = \`\${baseURL}\${path}\`.replace(/\\/\$/, \"\")

const res = await fetch(url, { \...init, headers, credentials:
\"include\" })

if (!res.ok) {

const text = await res.text().catch(() =\> \"\")

throw new Error(\`HTTP \${res.status} \${res.statusText}: \${text}\`)

}

if (res.status === 204) return undefined as unknown as T

return (await res.json()) as T

}

return {

get: \<T\>(p: string) =\> request\<T\>(p),

post: \<T\>(p: string, body?: unknown) =\>

request\<T\>(p, { method: \"POST\", body: JSON.stringify(body ?? {}) }),

patch:\<T\>(p: string, body?: unknown) =\>

request\<T\>(p, { method: \"PATCH\", body: JSON.stringify(body ?? {})
}),

del: \<T\>(p: string) =\> request\<T\>(p, { method: \"DELETE\" }),

}

}

export const http = createHttp()

**Instructions**

## **Step 8 --- Create tanstack query Hooks**

**File:** src/features/favorites/hooks.ts

import { useMutation, useQuery, useQueryClient } from
\"@tanstack-query\"

import { API, EntityType, type FavoriteItem, type Group, type Paginated
} from \"./api\"

// (Optional) centralize keys so they're consistent everywhere

const qk = {

groups: (type: EntityType) =\> \[\"favorite-groups\", type\] as const,

favorites: (type: EntityType, opts?: { groupId?: string; cursor?:
string; limit?: number; sort?: string }) =\>

\[\"favorites\", type, opts ?? {}\] as const, // use {} to keep the key
stable if opts is undefined

}

export function useFavoriteGroups(type: EntityType) {

return useQuery\<Group\[\]\>({

queryKey: qk.groups(type),

queryFn: () =\> API.listGroups(type),

})

}

export function useCreateGroup() {

const qc = useQueryClient()

return useMutation\<Group, Error, { name: string; type: EntityType }\>({

mutationFn: API.createGroup,

onSuccess: (group) =\> {

qc.invalidateQueries({ queryKey: qk.groups(group.type) })

},

})

}

export function useDeleteGroup(type: EntityType) {

const qc = useQueryClient()

return useMutation\<void, Error, string\>({

mutationFn: (groupId) =\> API.deleteGroup(groupId),

onSuccess: () =\> {

qc.invalidateQueries({ queryKey: qk.groups(type) })

},

})

}

export function useFavorites(

type: EntityType,

opts?: { groupId?: string; cursor?: string; limit?: number; sort?:
string }

) {

return useQuery\<Paginated\<FavoriteItem\>\>({

queryKey: qk.favorites(type, opts),

queryFn: () =\> API.listFavorites(type, opts),

})

}

export function useToggleFavorite(type: EntityType, groupId?: string \|
null) {

const qc = useQueryClient()

return useMutation\<

\| { removed: true }

\| { removed: false; created: FavoriteItem },

Error,

{ entityId: string; favoriteId?: string \| null }

\>({

async mutationFn(args) {

if (args.favoriteId) {

await API.removeFavorite(args.favoriteId)

return { removed: true as const }

}

const created = await API.addFavorite({

entityType: type,

entityId: args.entityId,

groupId: groupId ?? null,

})

return { removed: false as const, created }

},

onSuccess: () =\> {

qc.invalidateQueries({ queryKey: qk.favorites(type) })

qc.invalidateQueries({ queryKey: qk.groups(type) })

},

})

}

Each hook corresponds directly to one logical function in the backend.\

## **Step 11 --- Verify API Connections**

**Goal:** Confirm all frontend calls are wired to the correct backend
endpoints.

**Instructions**

1.  Run the app locally:

npm dev

2.  Open **Chrome DevTools → Network**.

3.  Interact with your screen (load, create, update, delete).

4.  Confirm that each request returns **200 or 201**.

5.  If you see CORS errors, verify your frontend origin in the backend
    configuration.

6.  For authentication errors (401 / 403), confirm your token handling
    logic.

## **Notes**

- The Query Provider and HTTP Client are created **once** per app.

- Each microservice has its own folder under src/features.

- Endpoint discovery and export are handled by the separate SOP titled\
  **"Finding and Exporting Endpoints from Microservices (with OpenAPI &
  Fallback Scan)."\**

- Storybook base components must **never** be modified --- copy only.

- Always reference **Figma** for toast, modal, and alert designs.

- Use consistent naming conventions (api.ts, hooks.ts, types.ts) for
  clarity.

- Maintain the same architectural structure for all future
  frontend--backend integrations.
