# **Production Deployment -- Promoting QA-Approved Code to Live**

**Project:** marketeq-projects-live\
**Audience:** Maintainers, Lead Developers, or Frontend Developers
handling releases

## **🎯 Purpose**

This guide walks you through how to take a feature that has already
passed QA testing and promote it to the **live site**
(marketeq-projects-live).\
You'll also learn how to confirm that the deployment worked and what to
do if you need to roll it back.

## **1️⃣ Confirm that QA has approved your feature**

Before you do anything, check that your feature was fully tested and
approved.

- In **Jira**, make sure the test case for your feature is marked **"QA
  Approved."\**

- Review any **Pumble DMs** from the QA tester confirming approval.

*This step ensures that you only release stable, verified code. Skipping
it can accidentally push bugs to production.*

## **2️⃣ Make sure your pull request (PR) is ready**

Open your PR in GitHub and confirm:

- There are **no unresolved comments**.

- The build checks in GitHub show green (✅).

- The preview site in marketeq-projects-frontend-testing is the same
  version that QA approved.

*This quick check prevents mistakes, like merging an older or broken
version.*

## **3️⃣ Merge your pull request into the main branch**

Once QA has approved and your PR looks good:

1.  Go to your PR in GitHub.

2.  Click **"Squash and merge"** or **"Merge pull request."\**

3.  Confirm the merge.

*When you merge, Vercel automatically starts building the new version on
the **live project** (marketeq-projects-live). You don't need to do
anything else to trigger the deployment.*

## **4️⃣ Wait for the live build to finish**

Open the **Vercel Dashboard → marketeq-projects-live → Deployments**.\
You'll see a new deployment start automatically. Wait until it shows
**"Ready ✅."**

*This means the live website has finished building and the new code is
now available to the public.*

## **5️⃣ Verify that the live site works correctly**

Once the build is complete, open the live site:\
👉
[[https://marketeq-projects-live.vercel.app]{.underline}](https://marketeq-projects-live.vercel.app/)

Check a few key things:

- Your feature or page appears correctly.

- Navigation, buttons, and forms work as expected.

- There are no console errors in your browser.

- The app connects to the **production API**, not the dev one.

*Think of this as a final smoke test --- it's your responsibility to
make sure the public site looks exactly like what QA approved.*

## **6️⃣ Notify the QA tester**

After confirming that the feature is live and stable:

1.  Open **Pumble**.

2.  Send a **direct message (DM)** to the QA tester who approved your
    feature.

3.  Use a short message like:\
    \
    \
    "Hey \[Name\], just letting you know your approved feature is now
    live on production:
    [[https://marketeq-projects-live.vercel.app]{.underline}](https://marketeq-projects-live.vercel.app/)
    🚀"

*This closes the loop so QA knows their approved work is officially
released.*

## **7️⃣ Roll back if something breaks**

If the live build has issues (e.g., major bugs, broken layout, or failed
integrations):

1.  Go to **Vercel → marketeq-projects-live → Deployments.\**

2.  Find the previous working deployment.

3.  Click **"Instant Rollback."\**

*This immediately restores the last stable version of the site.
Rollbacks are safe, fast, and should be used anytime a bad release
causes user problems.*

## **8️⃣ Let the team know about the rollback (if it happens)**

If you do roll back:

1.  DM the QA tester and the team lead in **Pumble** to explain briefly
    what went wrong.

2.  Add a note to the Jira ticket with what you observed and that you
    rolled back to a previous version.

*Clear communication keeps everyone aligned and helps QA re-test the
fixed build later.*

## **9️⃣ Prepare for the next sprint**

After a successful release:

- Close or move your Jira task to **"Done."\**

- Delete your feature branch (GitHub or local).

- Start fresh from main for your next feature.

*Keeping your branches clean makes future deployments smoother and
prevents confusion later.*

## **⚙️ Quick troubleshooting**

  **Problem**                         **What It Means**                        **How to Fix**
  ----------------------------------- ---------------------------------------- ----------------------------------------------------------------------------------------------
  **Vercel build failed**             Missing env variable or dependency.      Check the error log in Vercel → "Deployments" and fix the issue in the repo before retrying.
  **Site not updating after merge**   Cached DNS or browser data.              Hard-refresh (Cmd+Shift+R) or clear cache.
  **Feature missing from live**       Wrong branch merged or merge conflict.   Verify your PR went into main and re-merge if needed.
  **Live site looks broken**          Possibly a frontend build issue.         Roll back immediately, notify QA, and re-test your fix.

## **🚀 Final Notes**

- You don't need to deploy manually --- Vercel does it automatically
  once you merge to main.

- Always verify that you're on the **live project
  (marketeq-projects-live)**, not the testing one.

- Always DM QA privately in **Pumble** for release confirmations or
  issues --- never post deployment updates in group channels.
