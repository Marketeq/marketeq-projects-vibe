# **QA Testing & Approval Process -- Reviewing Features in Vercel Testing**

**Project:** marketeq-projects-frontend-testing\
**Audience:** QA Testers

## **🎯 Purpose**

This guide explains how to test new features that developers deploy to
the **Vercel testing project**, verify that they meet the requirements
in **Jira**, and communicate your results clearly.\
Your job is to make sure everything works as intended before the feature
moves to the live site.

## **1️⃣ Find the feature assigned to you**

Developers will notify you when their work is ready for testing.\
You'll find these tasks in **Jira**, usually in the "Ready for QA"
column, or you'll receive a **Pumble direct message (DM)** with the
preview link.

*Always check Jira first, but if you get a Pumble DM, that means the dev
is directly handing you their testing link.*

## **2️⃣ Open the preview build**

Each feature will include a preview URL that looks like this:

https://feature-name\--marketeq-projects-frontend-testing.vercel.app

Click the link to open the site in your browser.

*This is a private build of that specific feature. You'll be testing it
exactly how it will look before it goes live.*

## **3️⃣ Review the related Jira test case**

In Jira, find the test case linked to the feature.\
Read through the **description, acceptance criteria, and test steps** so
you know what needs to be verified.

*Always start with the test case --- it tells you what success looks
like, what actions to try, and what to expect.*

## **4️⃣ Test the feature end to end**

Go through every step in the test case carefully:

- Try all buttons, inputs, and navigation paths.

- Check the page on both desktop and mobile view.

- Make sure there are no console errors in your browser (open DevTools →
  Console tab).

- Confirm that any forms, popups, or submissions work as expected.

*Think of yourself as the first real user --- your goal is to break it
gently before our customers do.*

## **5️⃣ Verify data and connections**

If the feature interacts with APIs (like saving info, fetching user
data, or showing a project list), make sure those calls work correctly.\
You can open **Network** in your browser's DevTools to see if any
requests fail.

*Failed network requests or 500/400 status codes mean something broke
between the frontend and backend --- these need to be reported clearly.*

## **6️⃣ Test against the design and logic**

Compare what you see to the original **Figma design** or acceptance
criteria in Jira.

- Does spacing, text, and button alignment match?

- Are labels, icons, and colors consistent?

- Does the workflow make sense for real users?

*Small design mismatches can confuse users later, so flag them even if
the functionality works.*

## **7️⃣ Record your results**

When you finish testing, write down exactly what you observed:

- **If it passed:** Say what you tested and confirm it works.

- **If it failed:** Include the exact steps you took, what you expected,
  and what actually happened.

*Good QA notes save everyone time. A clear explanation means the
developer can fix it quickly without guessing.*

## **8️⃣ Report bugs or issues**

If something doesn't work:

1.  Go to the **same Jira test case**.

2.  Add a comment describing the bug.

3.  Include a short summary, steps to reproduce, expected result, actual
    result, and attach screenshots if possible.

4.  Set the ticket status to **"Needs Fix."\**

5.  Send a **Pumble DM** to the developer who submitted the feature ---
    just a short note like:\
    \
    \
    "Hey \[Name\], I finished testing your feature. I found a bug with
    \[brief description\]. I added the details in Jira."

*Always message them privately in Pumble so they see it quickly. Don't
post bugs in group channels.*

## **9️⃣ Approve the feature if everything passes**

If the feature meets all the requirements:

1.  In Jira, comment something like:\
    \
    \
    "Tested on Vercel preview. All acceptance criteria passed ✅"

2.  Change the status to **"QA Approved."\**

3.  Send a short **Pumble DM** to the developer, for example:\
    \
    \
    "Hi \[Name\], your feature passed QA. You can move forward with
    deployment 🚀."

*This clear handoff lets the dev know it's safe to promote the code to
production.*

## **🔟 Retest after developer fixes**

If a developer fixes a bug and asks you to retest:

1.  Click the new preview link (it updates automatically when they push
    changes).

2.  Recheck the areas that were broken before.

3.  If it's fixed, update the Jira ticket to **"QA Approved."\**

*Always confirm fixes on the same preview build before giving the green
light.*

## **⚙️ Helpful testing reminders**

  **Task**                                   **Why It Matters**
  ------------------------------------------ -----------------------------------------------------
  Use Chrome or Firefox for consistency      These browsers show the most accurate build results
  Clear cache before each test               Old data can cause false bugs
  Use both desktop and mobile views          We build responsive designs
  Test user flows, not just visuals          Logic errors are just as important as UI
  Keep your feedback friendly and specific   QA helps improve the product, not blame people
