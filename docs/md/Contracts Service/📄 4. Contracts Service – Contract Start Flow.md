# **📄 Contracts Service -- Contract Start Flow**

## **1. Purpose**

The start flow ensures that contracts only become **active** after the
**initial deposit** has been paid.

- **Pending:** Contracts are created but inactive.

- **Active:** Contracts begin billing + time tracking once deposit is
  cleared.

- **Atomicity:** If a project has 5 contracts, all 5 must activate
  together after the deposit.

## **3. Lifecycle: Pending → Active**

1.  **Group Created\**

    - Client submits a request with multiple talent (e.g., POST
      /contract-groups).

    - Contracts are created in PENDING state.

    - A contract.group.created event is emitted → Checkout Service
      generates a **bundled deposit session**.

2.  **Deposit Attempt\**

    - Client is redirected to Checkout Service to pay.

    - If payment is successful, Checkout emits checkout.deposit.cleared
      → Contracts Service webhook.

    - If payment fails, Checkout emits checkout.deposit.failed.

3.  **Activation Saga\**

    - On checkout.deposit.cleared:

      - Contracts Service sets depositPaid = true on the group.

      - All pending contracts in the group → ACTIVE.

      - Each contract gets startAtISO = now().

      - Events emitted: contract.activated (per person) +
        contract.group.activated.

    - This happens **atomically**: either all contracts in the group
      activate, or none do.

4.  **Failure Handling\**

    - If checkout.deposit.failed:

      - Contracts remain PENDING.

      - A retry window (e.g., 7 days) is enforced.

      - If not resolved, the group and its contracts are auto-CANCELED.

export class ActivationSubscriber {

async activateGroupContracts(groupId: string) {

// 1. Start DB transaction

// 2. Fetch all contracts in the group

const contracts = \[

{ id: \'c1\', status: ContractStatus.PENDING },

{ id: \'c2\', status: ContractStatus.PENDING }

\];

// 3. Update ALL contracts to ACTIVE

const now = new Date().toISOString();

contracts.forEach(c =\> {

c.status = ContractStatus.ACTIVE;

c\[\'startAtISO\'\] = now;

});

// 4. Commit transaction

// 5. Emit events

return contracts;

}

}

## **5. Error Handling**

- **Deposit failed immediately:** Contracts remain pending. Notify
  client.

- **Deposit retry window expired:** Auto-cancel all contracts in group.
  Emit contract.canceled.

- **Partial update attempts (edge case):** If one contract fails DB
  update, **rollback entire transaction** → no contracts activate.

## **6. Events Involved**

- **Emitted\**

  - contract.group.created → Checkout Service

  - contract.activated (per person) → Time Tracking + Notifications

  - contract.group.activated → Billing Service

- **Consumed\**

  - checkout.deposit.cleared → Activate contracts

  - checkout.deposit.failed → Retry/cancel flow
