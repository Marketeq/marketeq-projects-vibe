# **Backend -- Database Migrations & Versioning**

Purpose: Provide backend developers with clear instructions for managing
schema changes in the **Portfolio Service** using TypeORM migrations.
This ensures consistency across environments and avoids using
synchronize: true in production.

## **0) Prerequisites**

- Service path: apps/portfolio-service

- TypeORM configured in app.module.ts with synchronize: false

- Database URL set in .env / Render dashboard: DATABASE_URL

- Installed dev dependencies already include typeorm and ts-node-dev

## **4) Configure TypeORM for Migrations**

> File: ormconfig.js at service root (or inline in app.module.ts if you
> prefer).

î°ƒmodule.exports = {

type: \"postgres\",

url: process.env.DATABASE_URL,

entities: \[\"dist/\*\*/\*.entity.js\"\],

migrations: \[\"dist/migrations/\*.js\"\],

cli: {

migrationsDir: \"apps/portfolio-service/migrations\"

}

};

î°‚Ensure your build outputs migrations to dist/migrations/.

## **5) Generate Migrations (Local Dev)**

Run this command when you change entities:

î°ƒnpm run typeorm migration:generate \-- -n AddGalleryColumn -d
apps/portfolio-service/migrations

î°‚It will scan entity changes and create a migration file.

Or create manually:

î°ƒnpm run typeorm migration:create \-- -n CustomMigration -d
apps/portfolio-service/migrations

î°‚

## **6) Run Migrations (Local)**

**î°ƒ**npm run typeorm migration:run

î°‚To revert:

î°ƒnpm run typeorm migration:revert

î°‚

## **7) Run Migrations (Render Deploy)**

1.  Ensure migrations are built into dist/migrations/ by your CI/CD
    build step.

2.  In Render deploy settings, add a **pre-deploy command**:

î°ƒnpm run typeorm migration:run

3.  î°‚This ensures DB is up-to-date before starting the service.

4.  Confirm logs show migrations applied successfully.

## **8) Version Table**

TypeORM maintains a migrations table in Postgres automatically. It
tracks which migrations have been applied.

- Never edit old migration files once deployed. Create a new one for
  changes.

- Each migration = a new version step.

## **9) Best Practices**

- Keep synchronize: false in production to avoid accidental schema
  drift.

- Always review generated migrations before committing.

- Use semantic names (AddPersonaTable, AddTagsColumn) for clarity.

- Run migrations locally before pushing to main.

With this doc, backend developers can safely create, run, and manage DB
migrations for the Portfolio Service without relying on synchronize:
true. This ensures reliable schema versioning across environments.
