# **Backend -- Deployment & Ops Playbook**

Purpose: Provide a repeatable deployment + operations guide for the
**Portfolio Service** on Render. This ensures consistent environment
setup, scaling, and monitoring across the team.

## **0) Context**

- Service: apps/portfolio-service

- Runtime: Node.js 20 (as per Dockerfile)

- Database: Postgres on Render (managed)

- Deployment target: Render Web Service

## **1) Environment Variables (Render Dashboard)**

Set these under **Environment → Environment Variables** in Render:

- DATABASE_URL → Postgres connection string

- PORT → 3000 (optional; defaults to 3000)

- ALLOWED_ORIGINS → comma-separated list of frontend domains (e.g.,
  https://app.example.com,https://staging.example.com)

- JWT_SECRET or JWT_PUBLIC_KEY → auth key depending on JWT strategy

- CLOUDFLARE_ACCOUNT_ID → for media uploads

- CLOUDFLARE_ACCESS_KEY_ID

- CLOUDFLARE_SECRET_ACCESS_KEY

- CLOUDFLARE_BUCKET

- CLOUDFLARE_PUBLIC_URL → CDN URL for public access

- MAX_FILE_SIZE_MB → e.g., 10

**Best practice:** Store secrets in Render's environment variable
manager, never commit them.

## **2) Build & Start Commands**

- **Build Command**: npm ci && npm run build

- **Start Command**: node dist/main.js

These should match the Dockerfile configuration and your monorepo build
scripts.

## **3) Scaling Rules**

- Default: 1x instance (512 MB RAM)

- Enable **autoscaling** in Render settings:

  - **Min instances**: 1

  - **Max instances**: 3 (adjust as traffic grows)

  - Autoscale triggers on sustained CPU \>70% or RAM \>70%

- For heavy traffic periods, manually scale to more instances.

**DB scaling:**

- Start with Standard-256MB Postgres instance.

- Enable **high availability** and **automatic daily backups**.

- Scale vertically if query latency rises (\>100ms avg).

## **4) Monitoring & Logs**

### **Logs**

- Access logs from Render dashboard → Logs tab.

- Stream logs with Render CLI: render logs \<service-id\>.

### **Alerts**

- Enable **Render Health Checks**: /healthz endpoint should return 200.

- Configure alerting in your monitoring stack (PagerDuty, Opsgenie,
  Slack) to notify on:

  - Service crash/restart

  - Failed health checks

  - Migration errors at deploy

### **Metrics**

- Monitor Render metrics: CPU, RAM, instance count.

- Monitor Postgres metrics: connections, query time, slow queries.

### **Backups**

- Enable automated DB backups (daily snapshots).

- Retain at least 7--14 days for rollback.

- Test restores quarterly.

## **5) Deployment Workflow**

1.  Developer merges to main branch.

2.  CI/CD pipeline runs tests + builds service.

3.  Render auto-deploys new container.

4.  Pre-deploy command runs DB migrations:

npm run typeorm migration:run

5.  \
    Service restarts with new build.

## **6) Incident Response**

- If deploy fails:

  - Roll back via Render's "Rollback" button.

  - Check logs for failed migrations or missing env vars.

- If DB corruption suspected:

  - Restore from last backup.

  - Re-run migrations to catch up.

## **7) Best Practices**

- Never use synchronize: true in prod.

- Always run migrations in CI/CD before service boot.

- Keep secrets out of logs.

- Limit ALLOWED_ORIGINS to known domains.

- Rotate JWT secrets every 6--12 months.

This playbook ensures anyone on the team can deploy, scale, and operate
the Portfolio Service on Render without guesswork.
