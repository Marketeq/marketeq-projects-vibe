# **SOP --- Export & Enrich OpenAPI (Per Microservice) + How to Generate Frontend-Only Docs**

**Audience:** Backend engineers working on a single microservice\
**Goal:** Produce one file (openapi.json) **for your microservice
only**, enrich its descriptions, then use our template to generate
Stripe-style **frontend-only** docs grouped by epic/page.

## **0) Scope, Format, and Best-Practice Defaults**

**Explanation\**
We document **one microservice at a time** so engineers don't block each
other or export the whole platform. The exporter script lives **inside
your microservice** and writes a single openapi.json we can reuse for
docs, CI checks, and (later) a public API if we want. The frontend docs
will be "Stripe-style," epic-focused, and **hide internal/server-only
endpoints**.

**Decisions (best practices)**

- **Base URL & versioning (examples):** \${NEXT_PUBLIC_API_BASE_URL}/v1

- **Code snippets:** fetch with async/await (plain JS)

- **Auth header:** Authorization: Bearer \<token\> for protected routes

- **Frontend doc scope:** show endpoints intended for browser/client;
  **hide** internal/admin/server-only

## **1) Add the exporter script (per microservice)**

**File path\**
apps/\<microservice\>/scripts/export-openapi.ts (e.g.,
apps/contracts-service/scripts/export-openapi.ts)

**Explanation\**
This script boots **your microservice**, scans its controllers/routes
via NestJS Swagger, and writes an OpenAPI file for **this service
only**. We keep it in the repo **permanently** so it's easy to re-export
whenever endpoints change and to wire into CI.

**Code**

****// apps/\<microservice\>/scripts/export-openapi.ts

import { NestFactory } from \'@nestjs/core\';

import { SwaggerModule, DocumentBuilder } from \'@nestjs/swagger\';

import { AppModule } from \'../src/app.module\';

import { writeFileSync, mkdirSync } from \'fs\';

import { join } from \'path\';

(async () =\> {

const app = await NestFactory.create(AppModule, { logger: false });

const config = new DocumentBuilder()

.setTitle(\'\<Microservice Name\> API\')

.setDescription(\'OpenAPI export for frontend docs (this microservice
only)\')

.setVersion(\'1.0.0\')

.addBearerAuth()

.build();

const document = SwaggerModule.createDocument(app, config, {

// If you use functional routing, uncomment:

// deepScanRoutes: true,

});

const outDir = join(process.cwd(), \'openapi\');

mkdirSync(outDir, { recursive: true });

const outFile = join(outDir, \'openapi.json\');

writeFileSync(outFile, JSON.stringify(document, null, 2));

await app.close();

// eslint-disable-next-line no-console

console.log(\`✅ OpenAPI written to \${outFile}\`);

})();

**Hints**

- Make sure this script runs in the **microservice** directory; it
  imports that service's AppModule.

- Keep the file committed --- it's tiny, safe, and useful for every
  future change.

## **2) Ensure Swagger is available for your service**

**Explanation\**
If your microservice doesn't already depend on NestJS Swagger, add it
once. These are dev-time deps and won't bloat your runtime.

**Code**

****\# from repo root or service folder (use the workspace/package
manager you use)

pnpm add -D \@nestjs/swagger swagger-ui-express

**Hints**

- If you use Fastify instead of Express, use \@fastify/static instead of
  swagger-ui-express.

- You **do not** need to host Swagger UI to export JSON --- this script
  just generates the file.

## **3) Export your microservice spec**

**Explanation\**
Run the exporter for **your service**. This produces a single
openapi.json at apps/\<microservice\>/openapi/openapi.json.

**Code**

****\# from apps/\<microservice\>/

pnpm ts-node scripts/export-openapi.ts

\# Output: apps/\<microservice\>/openapi/openapi.json

**Hints**

- If the command can't find ts-node, run from repo root (pnpm -w ts-node
  \...) or add a service-local script in package.json.

- If the app bootstraps env vars, ensure minimal required env are set
  (e.g., a .env.example with safe defaults).

## **4) Enrich endpoint descriptions with mini-prompts (in code)**

**Explanation\**
Short, plain-language descriptions make the final frontend docs
**great**. Add these to your controllers/DTOs via Swagger decorators so
they flow automatically into the export.

**Code samples**

****// Example: annotate controller method

\@ApiOperation({ summary: \'Create Payment Intent\' })

\@ApiBearerAuth()

\@ApiBody({ description: \'Amount in cents and payment method id\' })

\@ApiResponse({ status: 201, description: \'Payment intent created\' })

\@Post(\'payments/intents\')

createIntent(@Body() dto: CreateIntentDto) { \... }

**Mini-prompts (paste into \@ApiOperation.description if you prefer)**

- One-line purpose: "Say what this does and when a user needs it (1
  sentence, plain English)."

- Auth: "Does it need Authorization: Bearer \<token\> or public?"

- Params: "List path/query params with type, required/optional, and
  realistic examples."

- Body: "Show a realistic JSON example of the request body and explain
  fields in 1 line."

- Success: "Show a typical 200/201 JSON response with realistic values."

- Errors: "List 2--3 common errors with HTTP code and short example
  JSON."

- Pagination/filtering: "If applicable, show exact query params and an
  example URL."

**Hints**

- Keep each description to **1--2 sentences**. Clear beats clever.

- Use realistic values (not lorem ipsum) --- that's what helps frontend
  devs most.

## **5) Re-export after enriching**

**Explanation\**
After you add/edit descriptions, export again so the openapi.json
includes your updates.

**Code**

****\# from apps/\<microservice\>/

**Hints**

  changes together.

- If multiple services changed, each service runs its own exporter ---
  one file per service.

## **7) Run the "Frontend-Only Docs" generator (how to use the template)**

**Explanation\**
Here's how **you** (or the docs owner) "train the AI" so the output
matches our Stripe-style frontend doc. You'll provide the OpenAPI file
**plus** a tiny UI Map for the epic/page so the doc can say where each
endpoint connects in the UI.

**Inputs required**

- apps/\<microservice\>/openapi/openapi.json

- **UI Map** (small text file): a few bullets mapping UI to endpoints
  for the **epic/page\**
  Example:

Epic: Checkout

\- Checkout page (step 1): listPaymentMethods, getCart

\- Checkout page (step 2): createPaymentIntent, confirmPayment

\- Order confirmation page: getOrderById

**Prompt to use with the template (paste as the instruction to AI)**

****You are generating frontend-only API docs from OpenAPI + a small UI
Map.

Output must be Stripe-style: for each endpoint show Purpose, Used in UI
(epic/page),

Method & URL, Auth, Params (path/query), Request body, Response, Common
errors,

and a single copy-paste JS fetch example. Use
\${NEXT_PUBLIC_API_BASE_URL}/v1 as base.

Exclude internal/admin/server-only endpoints. Group by the provided
epic/page.

Keep language plain, 1--2 sentences per field, realistic examples, no
backend internals.

**Hints**

- The **Template Example** (Document 2 --- to be written next) defines
  the exact card layout and grouping. You will copy that template text
  verbatim as the AI's "format to follow."

- For this SOP, just remember: **OpenAPI + UI Map + Template** →
  generates the final doc FE wants.

## **8) Filtering rules for a frontend-only doc**

**Explanation\**
We don't want to show server-only endpoints to frontend developers. Use
these rules to filter.

**Rules**

- **Include** endpoints intended for browser use: CRUD for public
  resources, authenticated user actions, search/listing, checkout flows.

- **Exclude** endpoints tagged or named internal, admin, service,
  webhook, cron, sync, admin.\*, or that require server-only secrets.

- **If uncertain**, include the endpoint in a separate **"Appendix ---
  Backend Only"** section for the docs owner to review.

**Hints**

- When in doubt, **hide** from the frontend doc; the full openapi.json
  always remains available in the repo.

## **9) CI guardrail (recommended)**

**Explanation\**
Prevent stale docs by failing CI if endpoints changed but the exported
file wasn't refreshed.

**Code**

****\# in microservice CI job

pnpm ts-node scripts/export-openapi.ts

git diff \--exit-code openapi/openapi.json

**Hints**

- This is optional but keeps code and docs in lock-step automatically.

- If the diff trips, re-commit the updated openapi.json.

## **FAQ (short + practical)**

**Q: Is this exporter one-time or permanent?\**
A: **Permanent.** It lives in the repo so we can re-export anytime, wire
it into CI, and (later) power a public docs site if we choose.

**Q: Do I have to document the whole platform?\**
A: **No --- only your microservice.** The script runs inside
apps/\<microservice\> and outputs that service's endpoints.

**Q: What if my service has internal/admin-only routes?\**
A: Keep them in OpenAPI for engineering, but **exclude** them from the
frontend doc (or move them to an "Appendix --- Backend Only" if we need
temporary visibility).

**Q: Who do I notify when ready?\**
A: Send a **Pumble DM** to the docs owner with the file path and epic
name. Don't post in group channels.
