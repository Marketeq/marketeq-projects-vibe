# **Frontend Deployment -- From Local Machine to QA (Testing Project)**

**Project:** marketeq-projects-frontend-testing\
**Audience:** Front-End Developers

## **🎯 Purpose**

This guide explains how to take the code you've written on your
computer, push it to GitHub, let Vercel build it for testing, and hand
it off to the QA team for review. Follow each step carefully so the QA
team can test your work smoothly and nothing breaks in the shared
environment.

## **1️⃣ Start with a clean local setup**

Before doing anything, make sure your local branch is up to date with
the main codebase.

git checkout main

git pull origin main

*This updates your files so you're building on the latest version of
the project instead of an outdated copy.*

## **2️⃣ Create your own branch**

Create a new branch for your feature or fix.

git checkout -b feature/\<short-name\>

*Branch names should describe what you're building, like
feature/talent-profile or fix/signup-error. This keeps your work
organized and separate from others.*

## **3️⃣ Install and test your build locally**

Before you send anything to Vercel, make sure it actually works on your
computer.

pnpm i

pnpm lint

pnpm build

pnpm dev

*Run these commands to install dependencies, check for errors, and
confirm the project builds. If something fails here, fix it before
moving on. It's faster to fix issues locally than wait for Vercel to
fail later.*

## **4️⃣ Commit and push your code to GitHub**

Once your feature works locally, save your changes and push them up.

git add .

git commit -m \"feat: added \<your-feature-name\>\"

git push -u origin feature/\<short-name\>

*Committing creates a snapshot of your work, and pushing sends it to
GitHub so Vercel can pick it up.*

## **5️⃣ Open a Pull Request (PR)**

Go to GitHub and open a **Pull Request**:

- **Base branch:** main

- **Compare branch:** your feature/\<short-name\> branch

*Think of a PR as asking, "Hey, here's my work --- can someone review
and test it?" It's required before anything goes live.*

## **6️⃣ Let Vercel build your preview**

As soon as the PR is created, Vercel automatically builds a preview in
the testing project:\
👉 **marketeq-projects-frontend-testing.vercel.app**

You'll see a message in your PR checks like:

✅ Preview Ready --- View Deployment

https://feature-name\--marketeq-projects-frontend-testing.vercel.app

*This preview is your personal test site. It's what the QA team will
use to review your feature.*

## **7️⃣ Verify the preview build yourself**

Click your preview link and make sure everything loads correctly.

- Check that pages and components appear as expected.

- Open the browser console to confirm there are no errors.

- If something doesn't look right, go back to your code, fix it, and
  push again --- Vercel will rebuild automatically.

*Doing this quick self-check prevents QA from wasting time on issues you
could catch easily.*

## **8️⃣ Submit your feature for QA testing**

Once your preview works, hand it off to the QA team.

### **🧩 If your feature already has a Jira test case**

1.  Open the related **Jira ticket** for that feature.

2.  Paste your preview URL in a comment or the testing field, for
    example:

✅ Ready for QA

Preview URL:
https://feature-name\--marketeq-projects-frontend-testing.vercel.app

GitHub PR:
https://github.com/MarketeqDigital/marketeq-frontend/pull/\<number\>

3.  Tag the assigned QA tester in the comment.

4.  Move the ticket to **"Ready for QA."\**

*This tells QA that your feature is live and ready for them to start
testing.*

### **💬 If you need to send it manually (using Pumble)**

If you can't attach it in Jira or just want to confirm directly:

1.  Open **Pumble**.

2.  Send a **direct message (DM)** to your assigned QA tester --- don't
    post in group channels.

3.  Use this short format:

🧪 QA Submission -- Ready for Testing

Feature: \[short feature name\]

Preview URL:
https://feature-name\--marketeq-projects-frontend-testing.vercel.app

PR Link:
https://github.com/MarketeqDigital/marketeq-frontend/pull/\<number\>

Notes: \[any details they should know\]

*Sending a quick DM keeps communication clean and makes sure the right
person gets your link.*

## **9️⃣ Wait for QA approval**

The QA tester will check your preview site, run test cases, and either:

- DM you in Pumble with feedback, or

- Comment in Jira marking it **QA Approved ✅\**

*Don't merge or edit anything in main until you get explicit approval
from QA.*

## **🔟 Once QA approves**

After you receive QA approval:

- Make sure there are no open comments or issues.

- Prepare to merge your PR following the "Live Deployment" document
  (Document 3).

*You'll promote your code from testing → live after this point.*

## **⚙️ Troubleshooting common build issues**

  **Problem**                           **Explanation**                                            **Quick Fix**
  ------------------------------------- ---------------------------------------------------------- -------------------------------------------------------------------------------
  **Build failed on Vercel**            Usually caused by missing dependencies or syntax errors.   Run pnpm build locally and fix any red errors before pushing again.
  **Environment variables not found**   The testing project is missing a variable.                 Check Vercel → Settings → Environment Variables and add what's needed.
  **Node version mismatch**             You're running a different Node version locally.           Make sure your local version matches what's set in Vercel (Node 18 or newer).
  **Image or CORS errors**              The image or API isn't allowed yet.                        Ask a lead to add the domain or CORS rule in the gateway config.
