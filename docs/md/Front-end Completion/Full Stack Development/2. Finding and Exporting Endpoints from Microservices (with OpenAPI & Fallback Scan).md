# **Finding and Exporting Endpoints from Microservices (with OpenAPI & Fallback Scan)**

## **Purpose**

This document provides a complete, production-ready SOP for identifying
and exporting backend API endpoints across the Marketeq microservices
environment.\
It is designed for full stack developers integrating completed frontend
screens with backend data.\
\
The system currently includes multiple microservices with endpoints
distributed across several directory levels --- some properly nested
under /apps/\<service\>/src, and others still residing in the root /src
directory or inside Docker containers.

This SOP ensures developers can accurately locate and export all valid
endpoints, even in cases where service organization is inconsistent,
without refactoring backend code.

## **Outcome**

By following this SOP, the developer will:

1.  Identify every active controller endpoint across all microservices.

2.  Generate an accurate OpenAPI schema for each service (when
    supported).

3.  Perform a fallback manual scan if OpenAPI generation fails.

4.  Safely document and export endpoints for frontend integration.

5.  Confirm endpoint validity with test requests before use.

6.  Avoid accidental duplication, deletion, or modification of backend
    files.

## **Developer Role Clarification**

This SOP applies to **full stack developers** performing both frontend
and backend integration tasks.\
\
The frontend UI for each screen is already complete.\
Your task is to connect it to the correct backend endpoints --- not to
reorganize or rewrite backend logic.

## **Step 0 --- Verify Access**

Before starting, confirm that:

1.  You have access to the main backend repository
    (marketeq-projects-nestjs).

2.  You can open each microservice folder under /apps or root /src.

3.  You have permission to run Node.js scripts and install dev
    dependencies locally.

4.  You are not editing any code inside other developer-owned
    microservices.

## **Step 1 --- Understand Folder Distribution**

In the current repo, backend endpoints can live in several possible
locations.\
Review these carefully before assuming any endpoint is missing:

**1. Standard location (expected):**

****apps/\<microservice-name\>/src/

controllers/

dto/

services/

main.ts

**2. Alternate root-level location:**

****src/

\<microservice-name\>/

controller/

service/

**3. Split-location microservices (hybrid):\**
Some services have code both in apps/\<service\>/src **and** in src/.\
Example:

- category-sync-service/src/app/category.controller.ts

- src/category-sync-service/category.controller.ts

**4. Docker container or e2e folder variants:\**
Occasionally, you'll find partial endpoint code in:

category-sync-service-e2e/

These are typically for tests or container builds and should **not** be
used for integration.

## **Step 2 --- Quick Identification Checklist**

When locating endpoints:

✅ Always look for files named:

- \*.controller.ts

- \*.module.ts

- \*.service.ts

- main.ts (for bootstrapping)

✅ Common patterns:

- \@Controller(\'...\') defines the base route.

- \@Get, \@Post, \@Patch, \@Delete decorators define methods.

✅ Nested folders:\
If you see folders like /dto, /entities, or /interface, these are for
data and should not be confused with endpoint definitions.

## **Step 3 --- Use the OpenAPI Export Script**

If your microservice supports OpenAPI, you can automatically export
endpoints without scanning manually.

**Instructions:**

1.  Navigate to the root of your NestJS repository.

2.  Create the file scripts/export-openapi.ts (if it doesn't exist).

3.  Add the following code:

import { NestFactory } from \'@nestjs/core\';

import { DocumentBuilder, SwaggerModule } from \'@nestjs/swagger\';

import { AppModule } from \'../apps/\<your-service\>/src/app.module\';

import { writeFileSync } from \'fs\';

async function bootstrap() {

const app = await NestFactory.create(AppModule);

const config = new DocumentBuilder()

.setTitle(\'\<Service Name\> API\')

.setDescription(\'Auto-generated OpenAPI schema for endpoint
discovery\')

.setVersion(\'1.0\')

.build();

const document = SwaggerModule.createDocument(app, config);

writeFileSync(\'./openapi-\<service\>.json\', JSON.stringify(document,
null, 2));

await app.close();

}

bootstrap();

4.  Replace \<your-service\> with the correct microservice name (e.g.,
    favorites-service, listings-service).

5.  Run the script:

npx ts-node scripts/export-openapi.ts

6.  A file like openapi-favorites.json will be generated in your root
    directory.

7.  Open it and review all paths and methods (GET, POST, etc.) --- these
    represent your active endpoints.

## **Step 4 --- Fallback Scan (When OpenAPI Fails)**

If a microservice does not generate an OpenAPI file correctly, use this
fallback scan.

**Manual Fallback Steps:**

1.  From the repo root, run the following command to list all
    controllers:

grep -R \"@Controller\" apps/ src/ \| cut -d: -f1 \| sort -u

2.  To view methods for a specific controller:

grep -R \"@(Get\\\|Post\\\|Patch\\\|Delete)\" apps/\<service\>/src

3.  If no results are found, also search the root /src directory:

grep -R \"@Controller\" src/

4.  If you find both locations (apps/\<service\>/src and
    /src/\<service\>), open both to confirm which is actively registered
    in the running app's main.ts.

**Validation Tip:\**
Open main.ts and check which module is being imported into the bootstrap
function --- that tells you which controllers are actually loaded at
runtime.

## **Step 5 --- Combine and Export Results**

Once you've located all endpoints for a service:

1.  Create a file named:

/docs/exported-endpoints/\<service\>-endpoints.json

2.  Copy all endpoints from your OpenAPI file or manual scan results.

3.  Format as follows:

{

\"service\": \"favorites-service\",

\"basePath\": \"/favorites\",

\"endpoints\": \[

{ \"method\": \"GET\", \"path\": \"/favorites\" },

{ \"method\": \"POST\", \"path\": \"/favorites\" },

{ \"method\": \"DELETE\", \"path\": \"/favorites/:id\" }

\]

}

4.  Push this file to your branch and include it in your PR for
    integration reference.

## **Step 6 --- Confirm Endpoint Validity**

Before using any endpoint in frontend code:

1.  Launch the backend locally (pnpm start \<service\>).

2.  Open a REST client (e.g., Postman or Thunder Client).

3.  Test one endpoint from the exported list.

4.  Confirm the following:

    - Response code: 200 or 201 for success.

    - Data shape matches your frontend types.

    - Auth headers or tokens are working.

## **Step 8 --- Organizing Your Findings**

When you complete endpoint discovery for a service:

1.  Save all exported results in /docs/exported-endpoints.

2.  Include both the .json file and any .txt notes from fallback

3.  Add a summary table in your PR showing:

    - Number of endpoints found

    - OpenAPI or fallback method used

    - Confirmed base path

    - Status (validated / needs review)

## **Step 11 --- Optional Advanced Script (Automated Scan)**

If you want to automatically compile all controller routes without
manual searches,\
add this Node.js script:

**File:** scripts/find-endpoints.ts

import fs from \'fs\';

import path from \'path\';

function findFiles(dir: string, pattern: RegExp, results: string\[\] =
\[\]) {

for (const file of fs.readdirSync(dir)) {

const full = path.join(dir, file);

const stat = fs.statSync(full);

if (stat.isDirectory()) findFiles(full, pattern, results);

else if (pattern.test(file)) results.push(full);

}

return results;

}

const appsDir = path.join(process.cwd(), \'apps\');

const srcDir = path.join(process.cwd(), \'src\');

const controllers = \[

\...findFiles(appsDir, /\\.controller\\.ts\$/),

\...findFiles(srcDir, /\\.controller\\.ts\$/),

\];

fs.writeFileSync(\'./docs/exported-endpoints/all-controllers.txt\',
controllers.join(\'\\n\'));

console.log(\'Found controllers:\', controllers.length);

Run:

npx ts-node scripts/find-endpoints.ts

This generates a docs/exported-endpoints/all-controllers.txt file
listing every controller in the repo.
