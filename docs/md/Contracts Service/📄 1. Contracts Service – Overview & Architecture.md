# **📄 Contracts Service -- Overview & Architecture**

## **1. Purpose & Scope**

The **Contracts Service** manages all **contracts between clients and
talent profiles**.\
Key rules:

- ✅ **All contracts are for individuals only.\**

  - A contract always links **one client → one talent profile.\**

  - Projects, services, teams, and jobs are **associations only**.

  - Example: Starting a project with 5 talent = 5 separate individual
    contracts.

- ✅ **Contracts can be grouped.\**

  - A ContractGroup represents multiple individual contracts linked to a
    project, service, team, or job.

  - Deposits and billing apply at the **group level**, but lifecycle
    (start/end) is always per **individual contract**.

- ✅ **Contract lifecycle is event-driven.\**

  - Starts as pending.

  - Moves to active only after deposit is paid (via Checkout Service).

  - Ends as ended when the client stops an individual contract (final
    billing + review triggered).

## **3. Service Dependencies**

The Contracts Service does **not handle billing, time tracking, or
reviews itself.** Instead, it references other microservices:

- **Checkout Service** -- validates deposit payments before contracts
  activate.

- **Billing Service** -- manages recurring billing (weekly, monthly,
  etc.) per project/service/team.

- **Time Tracking Service** -- hours are tracked by contractId
  (individual basis).

- **Review Service** -- triggered when an individual contract ends;
  collects feedback.

- **Notification Service** -- sends invites, reminders, review prompts,
  and dashboard alerts.

- **User Service** -- validates that client and talent IDs exist.

- **Listing Service** -- validates project/service/job references.

## **4. Event-Driven Lifecycle**

Contracts rely on **events** (pub/sub model).

### **Lifecycle Flow**

1.  **Create Contract(s)\**

    - Client starts project/service/team/job.

    - System creates **ContractGroup** + N **pending contracts** (one
      per talent).

    - Event emitted: contract.group.created → Checkout Service handles
      deposit.

2.  **Activate Contracts\**

    - Checkout Service sends checkout.deposit.cleared.

    - Contracts Service sets all pending contracts in group → active.

    - Events emitted: contract.activated (per contract) +
      contract.group.activated.

3.  **End Individual Contract\**

    - Client selects one talent to stop (can't end group at once).

    - Billing Service finalizes last invoice/credit.

    - Contract marked ended.

    - Events emitted:

      - contract.ended (to Review Service → 90-day review flow).

      - Notification → dashboard + email alerts.

4.  **Optional: Cancel Pending Contracts\**

    - If deposit not paid in time, group auto-cancels.

    - Contracts marked canceled.

    - Event emitted: contract.canceled.

## **6. Developer Notes**

- **Important:** Never store project-level contracts. Always **store
  individual contracts** and group them with ContractGroup.

- **Atomic activations:** When deposit is paid, **all contracts in the
  group activate together**.

- **Per-person endings:** Only one contract at a time can be ended (UI
  and backend match this).

- **Events are required.** Don't call Billing/Reviews directly; publish
  events so downstream services handle their part.
