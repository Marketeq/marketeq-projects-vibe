# **📦 CI/CD Pipeline Setup & Branching Strategy**

*Deployment Automation for Microservices & Frontend (Vercel)*

## **Overview**

This document outlines how to set up a robust, automated CI/CD pipeline
using **GitHub Actions (Free Tier)** for a monorepo backend (NestJS
microservices) hosted on **Render**, and a separate frontend repo
(Next.js) hosted on **Vercel**. It covers:

- GitHub Actions automation

- Render Docker deployment for microservices

- Vercel auto-deploy setup

- Branching strategy: development, staging, live

- GitHub branch protection rules

- Guardrails to avoid premature frontend/backend integration

- Zen Code Review flow

## **🔀 Git Branching Model**

Use three permanent branches:

+---------------+------------------------------------------------------+
| > **Branch**  | > **Purpose**                                        |
+---------------+------------------------------------------------------+
| > development | > Local development only. NOT pushed to remote.      |
+---------------+------------------------------------------------------+
| > staging     | > All tested code pushed here for QA. Automatically  |
|               | > deploys to Render.                                 |
+---------------+------------------------------------------------------+
| > live        | > Production-ready code. Automatically deploys to    |
|               | > Render (backend) and Vercel (frontend).            |
+===============+======================================================+

> ✅ Replace main with live for clarity.\
> 💡 Never push untested or unreviewed code to staging or live.

## **🚦 GitHub Branch Protection Rules**

Set up branch protection for staging and live:

+---------------------+---------------+------------+
| > **Rule**          | > **staging** | > **live** |
+---------------------+---------------+------------+
| > Require pull      | > ✅          | > ✅       |
| > request review    |               |            |
+---------------------+---------------+------------+
| > Require status    | > ✅          | > ✅       |
| > checks            |               |            |
+---------------------+---------------+------------+
| > Restrict direct   | > ✅          | > ✅       |
| > pushes            |               |            |
+---------------------+---------------+------------+
| > Allow auto-merge  | > ❌          | > ❌       |
+=====================+===============+============+

## **🛠 CI/CD Automation Using GitHub Actions**

**Backend (NestJS Monorepo to Render):**

- Docker-based multi-container deployment

- All microservices reside under /apps/ folder

- One Render app with separate services configured via render.yaml

**Frontend (Next.js to Vercel):**

- Vercel auto-deploys from live branch of /marketeq-projects repo

- GitHub integration with Vercel handles everything after PR is merged

## **🧪 Preventing Premature Integration Issues**

Developers **must not** test frontend/backend communication before:

- Backend APIs are tested locally and approved

- Routes are confirmed in backend API docs

- Frontend has .env.local configured correctly

- Developers are aware of expected ports, paths, and proxy behavior

> ✅ All frontend/backend integration should be tested **after** pushing
> to staging, not in isolation unless explicitly approved.
