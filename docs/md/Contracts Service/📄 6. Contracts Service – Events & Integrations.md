# **📄 Contracts Service -- Events & Integrations**

## **1. Purpose**

The **Contracts Service** is not a standalone system. It orchestrates
contracts across multiple domains --- deposits, billing, time tracking,
reviews, and notifications --- using an **event-driven architecture**.

This document defines:

- All **events emitted** by Contracts.

- All **events consumed** by Contracts.

- How Contracts integrates with other microservices.

## **2. Event Transport**

- Events are delivered via a **message broker** (e.g., RabbitMQ, Kafka,
  or Redis Pub/Sub).

- Every service publishes and subscribes using a **topic exchange**
  pattern.

- All events must be **JSON objects** with at least:

{

\"eventName\": \"string\",

\"timestamp\": \"ISO-8601 string\",

\"data\": { \... }

}



## **3. Events Emitted by Contracts**

### **3.1 contract.group.created**

- **When:** A client creates a group of contracts (pending state).

- **Purpose:** Triggers Checkout Service to generate a bundled deposit
  session.

- **Payload:\**

****{

\"eventName\": \"contract.group.created\",

\"timestamp\": \"2025-10-01T12:00:00Z\",

\"data\": {

\"groupId\": \"grp-123\",

\"ownerClientId\": \"cli-456\",

\"type\": \"project\",

\"contracts\": \[

{ \"id\": \"c1\", \"talentId\": \"t1\", \"rate\": 85, \"status\":
\"pending\" },

{ \"id\": \"c2\", \"talentId\": \"t2\", \"rate\": 90, \"status\":
\"pending\" }

\]

}

}



### **3.2 contract.activated**

- **When:** A single contract becomes active after deposit is paid.

- **Purpose:** Notifies Time Tracking + Notifications + Analytics.

- **Payload:\**

****{

\"eventName\": \"contract.activated\",

\"timestamp\": \"2025-10-01T12:30:00Z\",

\"data\": {

\"contractId\": \"c1\",

\"groupId\": \"grp-123\",

\"clientId\": \"cli-456\",

\"talentId\": \"t1\",

\"startAtISO\": \"2025-10-01T12:30:00Z\"

}

}



### **3.3 contract.group.activated**

- **When:** All contracts in a group have been activated atomically.

- **Purpose:** Triggers Billing Service to start group billing cycle.

- **Payload:\**

****{

\"eventName\": \"contract.group.activated\",

\"timestamp\": \"2025-10-01T12:30:00Z\",

\"data\": {

\"groupId\": \"grp-123\",

\"clientId\": \"cli-456\",

\"contractCount\": 5

}

}



### **3.4 contract.ended**

- **When:** A client manually ends one individual contract.

- **Purpose:** Triggers Billing finalization, Review lifecycle,
  Notifications.

- **Payload:\**

****{

\"eventName\": \"contract.ended\",

\"timestamp\": \"2025-11-15T09:00:00Z\",

\"data\": {

\"contractId\": \"c1\",

\"clientId\": \"cli-456\",

\"talentId\": \"t1\",

\"groupId\": \"grp-123\",

\"reasonCode\": \"SERVICE_NO_LONGER_NEEDED\",

\"endAtISO\": \"2025-11-15T09:00:00Z\"

}

}



### **3.5 contract.canceled**

- **When:** Deposit fails or expires → contracts never activate.

- **Purpose:** Cleanup + notify client + free resources.

- **Payload:\**

****{

\"eventName\": \"contract.canceled\",

\"timestamp\": \"2025-10-05T14:00:00Z\",

\"data\": {

\"groupId\": \"grp-123\",

\"contracts\": \[\"c1\", \"c2\", \"c3\"\],

\"reason\": \"deposit_failed\"

}

}



## **4. Events Consumed by Contracts**

### **4.1 checkout.deposit.cleared**

- **Source:** Checkout Service.

- **Purpose:** Move group contracts from pending → active.

- **Payload:\**

****{

\"groupId\": \"grp-123\",

\"paymentRef\": \"chk_abc123\"

}



### **4.2 checkout.deposit.failed**

- **Source:** Checkout Service.

- **Purpose:** Keep contracts pending; may cancel if not retried.

- **Payload:\**

****{

\"groupId\": \"grp-123\",

\"reason\": \"payment_declined\"

}



### **4.3 billing.finalized**

- **Source:** Billing Service.

- **Purpose:** Confirms last invoice/credit applied for ended contracts.

- **Payload:\**

****{

\"contractId\": \"c1\",

\"invoiceId\": \"INV-12345\",

\"finalAmount\": 1200.00,

\"creditApplied\": 100.00

}



### **4.4 timetracking.hours.approved**

- **Source:** Time Tracking Service.

- **Purpose:** Ensure all hours logged are reconciled before billing.

- **Payload:\**

****{

\"contractId\": \"c1\",

\"hours\": 42,

\"approvedBy\": \"cli-456\"

}



## **5. Integrations with Other Services**

  **Service**                 **Interaction**
  --------------------------- -------------------------------------------------------------------------------------------------------------------------------
  **Checkout Service**        Receives contract.group.created → Creates bundled deposit session. Sends checkout.deposit.cleared or checkout.deposit.failed.
  **Billing Service**         Starts billing on contract.group.activated. Finalizes invoice/credits on contract.ended. Sends billing.finalized.
  **Time Tracking Service**   Logs hours against contractId. Sends timetracking.hours.approved before billing finalization.
  **Review Service**          Receives contract.ended → Opens 90-day review window + reminders.
  **Notifications Service**   Listens for contract.activated, contract.ended, contract.canceled → Sends UI banners, emails, push notifications.
  **Analytics Service**       Listens for all lifecycle events to compute spend, hours, rehire rates, ratings.

## **6. Example Event Flow**

**Client starts project with 3 people → creates 3 contracts in one
group.**

1.  contract.group.created → Checkout generates bundled deposit link.

2.  checkout.deposit.cleared → Contracts activates all 3 → emits
    contract.activated (x3) + contract.group.activated.

3.  Billing starts recurring invoices for the group.

4.  Talent logs hours → Time Tracking approves → Billing invoices.

5.  Client ends Bob's contract → Contracts emits contract.ended.

    - Billing finalizes Bob's invoice.

    - Review Service starts Bob's review window.

    - Notifications alert client + Bob.

## **7. Developer Notes**

- Always emit **per-contract events** (e.g., contract.activated,
  contract.ended) even when part of a group.

- Group events (contract.group.created, contract.group.activated) are
  for **billing/checkout orchestration**.

- Never mix billing logic inside Contracts; only emit/consume events.

- Ensure all events have contractId or groupId for traceability.
