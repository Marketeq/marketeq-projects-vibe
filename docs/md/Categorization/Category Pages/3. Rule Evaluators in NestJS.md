# **Doc 3 --- Rule Evaluators in NestJS**

**"Tax Groups" Query Engine**

> Goal: Convert rules (from Strapi Taxonomy Groups) into actual queries
> against the marketplace database.\
> Each evaluator supports different entities (Projects, Services, Teams,
> Talent), ranking modes (Top-Rated, Trending, Newest), subtree scoping
> (L1 → L2 → L3), tag filters, deduplication, and fallback logic.

## **Step 1 --- Define Evaluator Interface & Types**

**Explanation\**
We need a common contract so every entity evaluator implements the same
run() method. This makes it easy for the Assembler (from Doc 1) to
request blocks of results without knowing entity-specific logic.

**Folder path\**
/apps/api/src/category-pages/evaluators/types.ts

**Code**

****// /apps/api/src/category-pages/evaluators/types.ts

export type EvaluatorArgs = {

uuids: { l1: string; l2?: string; l3?: string }

level: 1 \| 2 \| 3

rule: {

id: string

entity: \'projects\' \| \'services\' \| \'teams\' \| \'talent\'

kind: \'top-rated\' \| \'trending\' \| \'newest\'

limit: number

gridSize?: number

carousel?: boolean

manualEnabled?: boolean

tagFilters?: string\[\] // optional tags from Strapi

}

limit: number

}

export interface Evaluator {

run(args: EvaluatorArgs): Promise\<any\[\]\>

}

## **Step 2 --- Build an Evaluator Factory**

**Explanation\**
A central factory decides which evaluator to use based on the rule's
entity type.

**Folder path\**
/apps/api/src/category-pages/evaluators/factory.ts

**Code**

****// /apps/api/src/category-pages/evaluators/factory.ts

import { Injectable } from \'@nestjs/common\'

import { ProjectsEvaluator } from \'./projects\'

import { ServicesEvaluator } from \'./services\'

import { TeamsEvaluator } from \'./teams\'

import { TalentEvaluator } from \'./talent\'

import { Evaluator } from \'./types\'

\@Injectable()

export class EvaluatorFactory {

constructor(

private readonly projects: ProjectsEvaluator,

private readonly services: ServicesEvaluator,

private readonly teams: TeamsEvaluator,

private readonly talent: TalentEvaluator

) {}

forEntity(entity: \'projects\'\|\'services\'\|\'teams\'\|\'talent\',
kind: string): Evaluator {

switch (entity) {

case \'projects\': return this.projects.forKind(kind)

case \'services\': return this.services.forKind(kind)

case \'teams\': return this.teams.forKind(kind)

case \'talent\': return this.talent.forKind(kind)

default: throw new Error(\`Unsupported entity: \${entity}\`)

}

}

}

## **Step 3 --- Subtree Query Helpers**

**Explanation\**
Since L1 → L2 → L3 are hierarchical, queries need to filter by the
correct subtree. We centralize this logic.

**Folder path\**
/apps/api/src/category-pages/utils/subtree.ts

**Code**

****// /apps/api/src/category-pages/utils/subtree.ts

export function subtreeFilter(uuids: { l1: string; l2?: string; l3?:
string }, level: 1\|2\|3) {

if (level === 1) return { l1: uuids.l1 }

if (level === 2) return { l2: uuids.l2 }

if (level === 3) return { l3: uuids.l3 }

return {}

}

## **Step 4 --- Projects Evaluator (example with blend + Hugging Face hook)**

**Explanation\**
Implements rules for Projects (top-rated, trending, newest). Uses
subtree filter, optional tags, deduplication, and ranking blend. Hugging
Face model is plugged in for quality scoring.

**Folder path\**
/apps/api/src/category-pages/evaluators/projects.ts

**Code**

****// /apps/api/src/category-pages/evaluators/projects.ts

import { Injectable } from \'@nestjs/common\'

import { Evaluator, EvaluatorArgs } from \'./types\'

import { subtreeFilter } from \'../utils/subtree\'

import { ProjectRepo } from \'../../repos/project.repo\'

import { hfScore } from \'../../ml/hf\'

\@Injectable()

export class ProjectsEvaluator {

constructor(private readonly repo: ProjectRepo) {}

forKind(kind: \'top-rated\'\|\'trending\'\|\'newest\'): Evaluator {

switch (kind) {

case \'top-rated\': return { run: (a) =\> this.topRated(a) }

case \'trending\': return { run: (a) =\> this.trending(a) }

case \'newest\': return { run: (a) =\> this.newest(a) }

default: throw new Error(\`Unsupported kind: \${kind}\`)

}

}

private async topRated({ uuids, level, rule, limit }: EvaluatorArgs) {

const base = await this.repo.queryBySubtree(subtreeFilter(uuids, level),
rule.tagFilters)

for (const p of base) {

const blend =

(p.rating \|\| 0) \* 0.4 +

(p.completed_count \|\| 0) \* 0.15 +

(p.revenue \|\| 0) \* 0.15 +

(p.ctr \|\| 0) \* 0.1 +

(p.admin_boost \|\| 0) \* 0.1 +

(await hfScore(\'project-quality-v1\', p)) \* 0.1

p.\_\_score = blend

}

return this.dedupe(base.sort((a,b) =\> b.\_\_score -
a.\_\_score).slice(0, limit))

}

private async trending({ uuids, level, rule, limit }: EvaluatorArgs) {

const base = await this.repo.queryBySubtree(subtreeFilter(uuids, level),
rule.tagFilters)

for (const p of base) {

const recency = (Date.now() - new Date(p.created_at).getTime()) /
(1000\*60\*60\*24)

const trendingScore = (p.ctr \|\| 0) \* 0.4 + (p.impressions \|\| 0) \*
0.3 - recency \* 0.05

p.\_\_score = trendingScore

}

return this.dedupe(base.sort((a,b) =\> b.\_\_score -
a.\_\_score).slice(0, limit))

}

private async newest({ uuids, level, rule, limit }: EvaluatorArgs) {

const base = await this.repo.queryBySubtree(subtreeFilter(uuids, level),
rule.tagFilters)

return this.dedupe(base.sort((a,b) =\> +new Date(b.created_at) - +new
Date(a.created_at)).slice(0, limit))

}

private dedupe(items: any\[\]) {

const seen = new Set()

return items.filter(i =\> {

if (seen.has(i.uuid)) return false

seen.add(i.uuid)

return true

})

}

}

## **Step 6 --- Fallback logic**

**Explanation\**
If a rule returns fewer items than its limit, fallback is triggered:

1.  Retry without tag filters.

2.  Expand search to parent level (e.g., if L3 is too sparse, pull from
    its L2).

3.  If still empty → block is suppressed (better to hide than show a
    broken grid).

**Folder path\**
Inside each evaluator, add a fallback helper.

**Code (snippet)**

****private async withFallback(queryFn: () =\> Promise\<any\[\]\>,
limit: number, uuids: any, level: number) {

let results = await queryFn()

if (results.length \>= limit) return results

// Fallback 1: drop tag filters

results = await queryFn()

if (results.length \>= limit) return results

// Fallback 2: expand scope (e.g., L3 → L2)

if (level === 3 && uuids.l2) {

results = await this.repo.queryBySubtree({ l2: uuids.l2 })

} else if (level === 2 && uuids.l1) {

results = await this.repo.queryBySubtree({ l1: uuids.l1 })

}

return results.slice(0, limit)

}

## **Step 7 --- Testing**

**Checklist**

- Create a rule in Strapi: Projects, kind="top-rated", limit=6.

- L1 page payload includes correct block with 6 projects ordered by
  blend.

- L2/L3 subtree filters work: only return projects tagged to that
  subtree.

- Tag filters applied correctly.

- Deduplication: same project never appears twice.

- Fallback: if L3 too empty, results expand to L2.

- Carousel rules enforce minimum content (3× grid size) --- already
  handled in Assembler (Doc 1).

## **✅ Summary**

- Rule Evaluators turn Strapi Tax Groups into actual content lists.

- Each evaluator enforces: subtree scoping (L1 → L2 → L3), tag filters,
  ranking blends, deduplication, and fallbacks.

- Hugging Face hook included for ML scoring.

- Consistent interface across Projects, Services, Teams, and Talent.
