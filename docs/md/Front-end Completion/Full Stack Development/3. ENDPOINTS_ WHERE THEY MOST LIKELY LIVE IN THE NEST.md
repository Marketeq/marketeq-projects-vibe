# **ENDPOINTS: WHERE THEY MOST LIKELY LIVE IN THE NEST.JS REPO**

This repo currently has **two active code zones** for NestJS services:

1.  **Microservice zone (by service folder):\**
    apps/\<service\>/src/\*\*\
    Examples you actually have:\
    apps/admin-service, apps/affiliate-referral-service,
    apps/algolia-service, apps/api-gateway, apps/auth-service,
    apps/autocomplete-service, apps/content-moderation-service,
    apps/favorites-service, apps/invitations-service,
    apps/listings-service, apps/portfolio-service,
    apps/time-tracking-service, apps/user-service.

2.  **Root zone (legacy/mixed):\**
    /src/\*\* (single shared tree with multiple domains)\
    Notable domain folders you actually have here:\
    src/algolia, src/auth-google, src/auth-linkedin, src/auth,
    src/client, src/config, src/database, src/project, src/publish,
    src/sms, src/styles, src/talent, src/user, src/utils, plus root
    files like src/app.controller.ts, src/app.module.ts, src/main.ts.

There is also a separate service under the root:\
/category-sync-service/src/app/\*\* (with controllers, modules, etc.)

And a frontend folder:\
/favorites-ui (ignore for endpoints; this is UI).

**Key reality:** a given feature's endpoints might be under
apps/\<service\>/src **and/or** under the root src domain area (e.g.,
src/project, src/publish, src/talent, src/user). Always search **both**
zones.

## **QUICK DECISION TREE (DO THIS FIRST)**

1.  **Know the domain/service you want** (e.g., favorites, listings,
    portfolio, auth, project, publish, talent, user).

2.  **Check for a dedicated microservice** under apps/\<service\>:

    - If it exists → search apps/\<service\>/src/\*\* **and also** the
      root src/\*\*.

    - If it doesn't exist → search only the root src/\*\* (look for the
      matching domain folder).

3.  **Also check** if there's anything proxied via apps/api-gateway (if
    present in your architecture).

## **FAST SEARCH COMMANDS (COPY/PASTE)**

> Requirements: rg (ripgrep). If you don't have it, use the "Plain grep
> / PowerShell" section below.

### **A) Microservice zone (if the service folder exists)**

Replace \<service\> with one from apps/, e.g. favorites-service,
listings-service, etc.

rg -n \--hidden \--glob \'!\*\*/node_modules/\*\*\' \\

\'@Controller\\(\|@(Get\|Post\|Patch\|Put\|Delete)\\(\'
apps/\<service\>/src

****rg -n \--hidden \--glob \'!\*\*/node_modules/\*\*\' \\

## **WHAT TO LOOK FOR (HOW TO BUILD THE PATHS)**

Inside each controller file:

- \@Controller(\'base/path\') → the **base route**.

- \@Get(\'sub\'), \@Post(\'sub\'), \@Patch(\':id\'), etc. → **method
  paths**.

Combine them:

- \@Controller(\'favorites\') + \@Get() → GET /favorites

- \@Controller(\'favorites\') + \@Post() → POST /favorites

- \@Controller(\'favorites\') + \@Delete(\':id\') → DELETE
  /favorites/:id

- Nested example: \@Controller(\'publish/media\') + \@Get(\':id\') → GET
  /publish/media/:id

## **RULES OF THUMB (SO NOBODY PANICS)**

- **Always search both zones.** Assume a split until proven otherwise.

- **Controllers + Modules = live routes.** If a controller isn't
  registered in a module, it won't serve traffic.

- **Don't refactor.** This is a locate-and-document task only.

- **Frontend doesn't fix structure.** The goal is to integrate now;
  cleanup can be a later project.
