# **📄 Contracts Service -- Data Model & Entities**

## **1. Purpose**

This document defines the **database schema** and **entity models** for
the Contracts Service.\
Every contract is always **individual**, but contracts can be grouped
under a project, service, team, or job for **bundled deposits and
billing**.

## **4. ContractGroup Entity**

File: /apps/contracts-service/src/entities/contract-group.entity.ts

import { Entity, PrimaryGeneratedColumn, Column, CreateDateColumn,
UpdateDateColumn } from \'typeorm\';

import { GroupType } from \'./enums\';

/\*\*

\* ContractGroup

\* Represents a logical grouping of individual contracts

\* (e.g., one project with multiple people hired).

\* Used for bundled deposits and billing purposes.

\*/

\@Entity(\'contract_groups\')

export class ContractGroup {

\@PrimaryGeneratedColumn(\'uuid\')

id: string;

// What type of group this is (project, service, team, or job)

\@Column({ type: \'enum\', enum: GroupType })

type: GroupType;

// The client who owns this group of contracts

\@Column()

ownerClientId: string;

// Optional references to external services (validate via Listing
Service)

\@Column({ nullable: true })

projectId: string;

\@Column({ nullable: true })

serviceId: string;

\@Column({ nullable: true })

teamId: string;

\@Column({ nullable: true })

jobId: string;

// Flag set true when initial deposit is successfully paid

\@Column({ default: false })

depositPaid: boolean;

// Automatic timestamps

\@CreateDateColumn()

createdAt: Date;

\@UpdateDateColumn()

updatedAt: Date;

}

## **5. Contract Entity**

File: /apps/contracts-service/src/entities/contract.entity.ts

import { Entity, PrimaryGeneratedColumn, Column, CreateDateColumn,
UpdateDateColumn } from \'typeorm\';

import { ContractStatus } from \'./enums\';

/\*\*

\* Contract

\* Always represents one individual talent profile hired by one client.

\* Can be linked to a project/service/team/job through its group.

\*/

\@Entity(\'contracts\')

export class Contract {

\@PrimaryGeneratedColumn(\'uuid\')

id: string;

// Reference to the parent group (for deposit & billing orchestration)

\@Column()

groupId: string;

// Core identifiers

\@Column()

clientId: string;

\@Column()

talentId: string;

// UI-driven fields (set during \"Hire Me\" flow)

\@Column()

title: string; // e.g., \"UX Designer for Mobile App\"

\@Column(\'decimal\', { precision: 10, scale: 2 })

rate: number; // hourly rate depending on contract

\@Column()

schedule: string; // e.g., \"full-time\", \"part-time\", \"custom\"

\@Column()

duration: string; // e.g., \"3 months\", \"6 months\", \"ongoing\"

// Lifecycle status

\@Column({ type: \'enum\', enum: ContractStatus, default:
ContractStatus.PENDING })

status: ContractStatus;

// Timestamps for lifecycle transitions

\@Column({ nullable: true })

startAtISO: string;

\@Column({ nullable: true })

endAtISO: string;

// Optional direct associations (duplicate for fast lookup)

\@Column({ nullable: true })

projectId: string;

\@Column({ nullable: true })

serviceId: string;

\@Column({ nullable: true })

teamId: string;

\@Column({ nullable: true })

jobId: string;

// Automatic timestamps

\@CreateDateColumn()

createdAt: Date;

\@UpdateDateColumn()

updatedAt: Date;

}

## **6. Association Rules**

- A **Contract** is always **1:1 client ↔ talent**.

- A **ContractGroup** may hold many contracts.

- Associations:

  - projectId, serviceId, teamId, jobId are **references only**
    (validated via Listing Service, not owned by Contracts).

  - If you hire 5 people for one project → you get 1 ContractGroup + 5
    Contracts.

- Ending a contract only ends that **one person**. The group remains
  active until all contracts end.

## **7. Status Transitions & Audit Trail**

### **States**

- **Pending** → created, waiting for deposit.

- **Active** → deposit confirmed (all contracts in group activate).

- **Ended** → contract ended manually for one person.

- **Canceled** → never activated (deposit failed/expired).

### **Rules**

- **Activation** is atomic at the group level: all pending contracts in
  a group move to active together once deposit is paid.

- **Endings** are individual: one person at a time.

- **Audit Trail**:

  - Use createdAt, updatedAt, startAtISO, endAtISO.

  - Every state change is published as an event (contract.activated,
    contract.ended, contract.canceled).
